Design philosophy

Most AI assistants are prompt wrappers — you send text, you get text back, with no memory of who you are or what you've discussed before. IO is architecturally different. Before any LLM sees your input, IO has already classified the intent of your query — deterministically via keyword dispatch for simple queries, or via an LLM planner for complex multi-tool tasks. A retrieval loop with self-assessment evaluates whether the recovered context is sufficient before synthesis; if not, IO reformulates and retries. Only then does the LLM receive an enriched, verified prompt.

After the LLM responds, ALMA evaluates the output against six principles — Care, Clarity, Limit, Awareness, Pragmatism, Silence. These are not safety filters. They are the grammar of a different kind of intelligence: conditions for IO's own coherence when operating in clinical contexts where ambiguity causes harm. ALMA evaluates each principle individually, cites the specific text that violates it, and explains why — evidence-based reasoning, not tone classification. If a violation is found at medium or high severity, IO rewrites with targeted corrections. Every correction logs to alma_training.jsonl — structured reasoning pairs for future LoRA distillation.

The LLM — whether local (qwen3:14b via Ollama), OpenAI, or Anthropic — is a replaceable component. IO is the layer that persists across models, sessions, and interactions. That is the architectural bet behind this project.

Execution flow

How a single query moves through IO

Every input passes through intent classification, tool execution, LLM generation, and ethical verification — in that order. No step is skipped. Click nodes to explore.

Loop logic: If answer = None after respond_node and step < max_steps, the graph cycles back to dispatch. Typical web+RAG queries run 2 cycles: cycle 1 fetches web context and returns answer=None, signaling the need for personal data; cycle 2 cross-references web results with ChromaDB and produces the final answer.

Deterministic vs LLM paths: time queries, ingest operations, and system queries never touch an LLM — they execute deterministically in dispatch_node and return directly. Only queries requiring reasoning or synthesis reach respond_node and the LLM.
